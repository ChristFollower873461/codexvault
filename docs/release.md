# Release and packaging status

This document records the current release and packaging status. It should stay more precise than marketing copy.

## Release bar

CodexVault should not be described as a trusted public binary release unless all of the following are true:

- code review is complete
- lint, release-gate e2e flows, frontend tests, Rust tests, and desktop smoke pass
- manual QA is complete
- platform packaging has been validated on the target OS
- binaries are signed
- macOS builds are notarized where applicable

## Current state

- `macOS`: source-build workflow and local package build verified.
- `Windows`: CI desktop smoke coverage is in place, but this repository does not yet claim a manually release-validated package.
- `Linux`: not a claimed initial-release target.
- `npm run tauri:build` currently succeeds for local macOS packaging.
- Produced macOS artifacts are still unsigned and unnotarized.
- No auto-update channel is configured.
- No external security audit has been completed.
- A dedicated release workflow now exists at `.github/workflows/codexvault-release.yml`.

## Release workflow modes

Run release from GitHub Actions using `CodexVault Release` (`workflow_dispatch`):

- `release_channel=preview` produces unsigned preview artifacts and `release-truth.json` marked untrusted.
- `release_channel=partial` requires signed + notarized macOS artifacts while allowing Windows unsigned preview artifacts.
- `release_channel=trusted` enforces:
  - signed + notarized macOS artifact
  - signed Windows artifact
  - non-empty Windows manual validation report input

If any required condition for the chosen channel fails, the workflow fails.

## Required secrets by mode

`partial` and `trusted` require Apple secrets:

- `CODEXVAULT_APPLE_CERTIFICATE_BASE64`
- `CODEXVAULT_APPLE_CERTIFICATE_PASSWORD`
- `CODEXVAULT_APPLE_SIGNING_IDENTITY`
- `CODEXVAULT_APPLE_ID`
- `CODEXVAULT_APPLE_APP_SPECIFIC_PASSWORD`
- `CODEXVAULT_APPLE_TEAM_ID`

`trusted` additionally requires Windows signing secrets:

- `CODEXVAULT_WINDOWS_CERTIFICATE_BASE64`
- `CODEXVAULT_WINDOWS_CERTIFICATE_PASSWORD`

If required secrets are missing, release for that channel is intentionally blocked.

## Artifact truth outputs

The release workflow emits:

- `release-truth.json` generated by `scripts/release/verify-release-truth.mjs`
- `SHA256SUMS` generated by `scripts/release/generate-checksums.mjs`

These files are the source of truth for whether artifacts are trusted.

## Release checklist

1. Run the verified command set above.
2. Complete [manual-qa.md](./manual-qa.md).
3. Choose release mode intentionally:
   - `preview` for all-preview artifacts
   - `partial` for trusted macOS + Windows preview
   - `trusted` for trusted cross-platform
4. For `trusted`, complete [windows-validation.md](./windows-validation.md) and provide report text in `windows_validation_report`.
5. Confirm `release-truth.json` fields match channel intent.
6. Confirm `README.md`, `SECURITY.md`, and this file still match reality.
7. Confirm no telemetry, remote content, or automatic config mutation was introduced.
8. Confirm export templates and demo-vault output still reflect the supported product surface.

## Artifact truth

Use this exact labeling policy:

- `preview` channel: unsigned/unnotarized preview artifacts, source review recommended.
- `partial` channel: `macOS (Signed + Notarized)` and `Windows (Unsigned Preview / Early Access)`.
- `trusted` channel: trusted cross-platform only when both macOS and Windows checks pass.

Do not label artifacts as fully trusted cross-platform unless `release-truth.json` reports `trustTier=trusted-cross-platform`.
